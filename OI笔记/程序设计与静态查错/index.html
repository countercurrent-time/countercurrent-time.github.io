程序设计与静态查错

发表于 1970-01-01 更新于 2020-10-01 分类于 OI笔记
在算法竞赛中写出简洁高效的代码。

编程范式

  • 面向过程：POP(Process Oriented Programming)，“面向过程”是一种是事件为中心的编程思想。就是分析出解决问题所需的步骤，然后用函数把这写步骤实现，并按顺序调用。
  • 面向对象：OOP(Object Oriented Programming)，用线性的思维。与面向过程相辅相成。在软件开发过程中，宏观上，用面向对象来把握事物间复杂的关系，分析系统。微观上，仍然使用面向过程。“面向对象”是以“对象”为中心的编程思想，即以封装、继承、类为核心。
  • 函数式编程：FP(Functional Programming)，把运算过程尽量写成一系列嵌套的函数调用，函数是“一等公民”，只使用表达式而不是语句。函数的功能不是封装操作，而是单纯地运算并返回确定的值，即没有”副作用”，避免耦合关联。变量只保存状态。

以上 3 种编程范式是应用最广的编程范式，也比较容易运用到 OI 当中。

面向过程编程

  1. 自顶向下，逐步求精、分而治之的编程。
  2. 迭代的速度往往比递归快很多，而 C++ 编译器在不开 O2 的情况下不提供尾递归优化。
  3. 缺点：没有面向对象易维护、易扩展；迭代虽然效率较高，开销较小，但是代码编写、调试和维护的复杂程度比函数式编程高。

面向对象编程

  • 自底向上的编程。
  • 面向对象是将事物高度抽象化。面向对象必须先建立抽象模型，之后直接使用模型就行了。
  • 优点：易维护、易调试、易扩展，由于面向对象有封装、继承、多态等特性，可以使代码更加易用、易于调试，非常适合。
  • 缺点：深层封装、过多的保护机制可能会导致难以使用和程序效率降低，而需要根据实际用途重新设计类，否则会破坏类的封装。

  • 面向对象编程是一个极其糟糕的主意，只有硅谷里的人能干出这种事情。

    — Edsger Dijkstra

算法竞赛中，如果同一份代码使用的算法或数据结构较多，我们可以把不同的算法或数据结构装进不同的名称空间或者类里。

函数式编程

  • 函数式语言中大量运用函数的思想，是面向数学的抽象，更接近自然语言，代码会比较简洁，也更容易理解。

  • 函数式编程依赖递归，开销大（不过函数式编程语言编译器通常会有很优秀尾递归优化），但递归调用深度为 $O(\log n)$的函数建议使用递归，因为可以缩减代码，而且 NOI 系列比赛中评测时开无限栈，所以不用担心栈溢出。

    （由于变量不可变，纯函数编程语言无法实现循环，这是因为 for 循环使用可变的状态作为计数器，而 while 循环或 do-while 循环需要可变的状态作为跳出循环的条件。）

  • 函数的抽象程度可以很高，所以函数式的代码可以更方便的复用。

  • NOI 系列比赛中不能使用 C++11 的特性，所以不能使用 lambda 表达式这个好东西。

  • C++ 中对函数式编程的支持仍然比较差。

  • 函数式编程中复制数据的开销会很大，且 C++ 编译器中不提供相关优化，所以该传指针还是要传指针。

函数式编程的思想运用到 fhq Treap、可持久化线段树等代码的编写中，会使这些数据结构容易理解很多，也可以避免传引用中意外产生的 BUG，同时弥补面向对象的封装导致的不足，并减少代码细节和出错的可能性。

算法竞赛中的代码

算法竞赛中，由于时间有限，质量要求高，可维护性要求低，OI 中不能使用事先准备的类与接口（ACM 可以），代码健壮性要求低，所以算法竞赛中，好的代码要求如下：

对于同一份可做题，一份好的代码定义为：保证程序能在标准评测机上通过，在可读性、易维护性较好的基础上，使用同一种格式化标准格式化代码后，代码长度尽可能短的代码。

算法竞赛中的代码应尽量做到简洁、高效。而且简洁往往比高效更重要。简洁的代码具有编程时间短，方便阅读与调试，细节十分少，不易出错等优点。并且简洁代码的思想不仅仅可以应用于模拟类题目。

算法竞赛的核心在于算法思想，而不是程序效率。当代码中算法的时间复杂度与正解的时间复杂度相同，且可以通过题目时，不应该过多注重代码优化，而应该尽可能缩短代码长度，多使用 STL 以使程序清晰易懂、且容易发现错误，减少 BUG 的产生。

算法库

构建模板算法的库，使自己可以方便地调用。

优点：

  • 用于归纳新学习的算法。
  • 听例题时，3分钟内按照自己的想法完成题目代码，验证自己的思路。
  • 做题时，更多地将注意力集中于算法的设计与思考，而不是程序实现的细节，以高效完成代码。
  • 用于对拍。
  • 比赛前用于复习模板。

事实上，构造算法库不是简单地完成模板。善于利用编程技巧，可以更快地实现算法库。

比如，在实现平衡树时，写一个二叉查找树的基类，然后 Treap、Splay、替罪羊树等类继承这个基类。这样实现的代码量比把3种平衡树完整码完少很多。

最优秀的调试方法：避免引入BUG。

有效的程序员不应该浪费很多时间用于程序调试，他们应该一开始就不要把故障引入。

调试方法

  1. 手推样例，理解题意。
  2. 手玩小样例。
  3. 静态查错
  4. 橡皮鸭调试法：对着自己喜欢的人或事物描述代码的思想算法与实现，可以快速找出程序的 BUG。在实际使用时，如果代码思路清晰，且搭配静态查错，效率往往非常高。
  5. 对拍。
    可以先打暴力。
    对于暴力，如果打不出正解，最好造大数据测试暴力用时。
  6. 过样例可能全 WA，也可能模拟赛爆零。要仔细检查。
  7. 下载测试数据。
  8. 与标准程序对拍。
  9. 与标准程序逐个函数对比，逐行对比。

使用 gdb 可以比较方便地调试，但 gdb 也有一些缺点：

  1. 学习成本较高。
  2. 每次修改代码后，再次运行时都要重复设置断点等操作。
  3. 输入命令耗时较多。
